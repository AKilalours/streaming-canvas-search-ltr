## What Is StreamLens?

StreamLens is a **Netflix-grade two-stage search and recommendation system** built from scratch. It models the exact pipeline used by Netflix, Spotify, and LinkedIn вҖ” candidate retrieval вҶ’ learning-to-rank reranking вҶ’ real-time serving with a multilingual GenAI explanation layer.

**Headline numbers:**
- LTR nDCG@10 = **0.9300** after fine-tuning e5-base-v2 on domain data
- **21 ML algorithms** across retrieval, ranking, personalization, and causal inference
- **106 API endpoints** covering search, explanation, feed, VLM, causal, and self-healing
- **44 languages** with GPT-4o-mini explanations in pure target script
- **RAGAS evaluation**: Faithfulness=0.705 В· Answer Relevance=0.752 В· Context Recall=1.000 вң…
- **HyDE query rewriting** for semantic queries via GPT-4o-mini hypothetical document generation
- **Faster-Whisper edge pipeline** вҖ” local ASR вҶ’ retrieval вҶ’ Llama3 at zero API cost
- Evaluated on MovieLens (150 queries) and independently validated on BEIR NFCorpus (323 medical queries, above published reference)

## Interview Prep

### Design a RAG for 1M PDFs вҖ” latency < 1.5s

> Clarify first: query vs indexing latency? p99 or average?

Stage 1 вҖ” Retrieval: BM25 + fine-tuned e5 вҶ’ 500 candidates in ~200ms (same pattern as StreamLens).
Stage 2 вҖ” Reranker: cross-encoder вҶ’ top-20 in ~300ms.
Stage 3 вҖ” LLM: GPT-4o-mini with top-5 chunks вҶ’ answer in ~700ms.
Total: ~1.2s. Cache on (query_hash, chunk_ids) in Redis вҶ’ p50=2.67ms on repeats.

**Where latency hides:** tokenization (50ms), FAISS search (100ms), reranker (300ms), LLM (700ms).

### Deploy LLM with smallвҶ’big routing and cost guardrails

Routing: complexity score < 0.4 вҶ’ Llama3 local ($0). > 0.4 вҶ’ GPT-4o-mini ($0.0008). > 0.8 вҶ’ GPT-4o ($0.008).
Guardrail: Redis counter per user per day. Hard cap $0.10/user/day. Throttle to local after cap.
Fail-open: GPT-4o-mini down вҶ’ Ollama Llama3 вҶ’ template. Never return empty.

### Make it resilient to data drift

Eval gates: 9 quality gates before promotion. nDCG drift > 5% вҶ’ alert + block.
Shadow mode: new model runs parallel, logs scores without serving. Beat prod by 2% for 24h вҶ’ A/B.
Rollback: Metaflow artifact versioning. Previous model always retained.
Drift found: 24.6% gap in pre-2010 content вҖ” quantified and roadmapped.

### Why BM25 + Dense instead of Dense-only?

BM25 wins on exact-match queries (title search, specific film names). Dense wins on semantic/mood queries ("something sad but funny"). Hybrid Оұ=0.2 measured to be optimal on this corpus вҖ” BM25-dominant because titles are short and exact. Neither alone reaches the hybrid nDCG.

### How did you prevent train/test leakage in LTR?

80/20 split by query_id, not by document. All qrels generated from held-out queries only. BM25 and dense scores computed fresh on test set. No document-level split which would leak through co-watch pairs.

### How does RAGAS evaluation work here?

Standard RAGAS uses OpenAI embeddings for semantic scoring. StreamLens uses the same sentence-transformer model already in the pipeline (all-MiniLM-L6-v2) for cosine similarity scoring вҖ” faithfulness measures if the explanation is semantically grounded in retrieved sources, answer relevance measures if it addresses the query. Context recall measures if retrieval surfaces content-rich documents. All 15 queries evaluated, all 3 targets met.

## What I Would Build Next

**1. ALS Collaborative Filtering (4th retrieval signal)**
The 1.29M co-watch pairs from PySpark can train a Matrix Factorization model. Adding ALS co-embeddings as a 4th retrieval signal would significantly improve cold-start recall.

**2. Real-Time FAISS Update via Flink**
Currently new content appears after the next batch job. A Flink consumer reading from Kafka could update the FAISS index within 60 seconds of new content being added.

**3. Temporal Drift Fix**
Analysis shows pre-2010 content scores 24.6% lower than post-2010 вҖ” sparse metadata. LLM-based metadata enrichment for older films closes this gap.

**4. LambdaRank Hyperparameter Tuning**
LightGBM defaults were used. Formal grid search would likely push nDCG@10 above 0.95.

**5. Online A/B Validation**
Current A/B test is offline simulation (p=0.065, underpowered). Real online experiment with 1,000 users would provide statistical significance.

**6. Hard Negative Mining for Fine-tuning**
Current fine-tuning uses random negatives. Hard negatives (near-miss films from the current model) would improve embedding quality substantially вҖ” estimated +0.03 dense nDCG.
